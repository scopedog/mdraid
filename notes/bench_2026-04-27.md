# Benchmark — kmec vs raid6 (2026-04-27)

After today's kmec features landed (sub-stripe RMW, degraded reads + writes,
scrub, hot-replace, small-write RMW, per-stripe parallel I/O, cross-stripe
pipeline), benchmark them against stock raid6 on the same loop geometry.

## Setup

- 6× 256 MiB sparse-file loop devices (k=4 data + 2 parity), 64 KiB chunk.
- Each VM is a bhyve guest sharing the same underlying virtio disk for all
  loops, so per-disk physical parallelism is **not** measured here — these
  numbers reflect kernel-side EC work, lock/scheduling overhead, and stripe
  pipelining.
- fio 3.36 with `--direct=1 --ioengine=libaio --iodepth=8 --time_based=1
  --runtime=8s` per workload, 64 MiB working set.
- `--assume-clean` for all raid6 arrays (so initial sync isn't part of the
  cost). kmec auto-skips initial sync via `mddev->recovery_cp = MaxSector`.

## rd07 (GFNI Raptor Lake bhyve guest, kernel 6.12.0-124.8.1.el10_1)

| Workload                    | bs   | A. stock raid6     | B. raid6 + raid_isal (GFNI) | C. kmec (k=4, m=2) |
|-----------------------------|------|-------------------:|----------------------------:|-------------------:|
| seq write (full-stripe)     | 256K | 1664 MB/s          | 1626 MB/s                   | **2183 MB/s** (+31% vs A) |
| seq read                    | 256K | 3106 MB/s          | **3277 MB/s** (+5%)         | 3172 MB/s          |
| rand write (sub-stripe RMW) | 4K   | **87 MB/s**        | 84 MB/s                     | 39 MB/s (−55%)     |
| rand read                   | 4K   | 141 MB/s           | **156 MB/s** (+11%)         | 127 MB/s           |
| scrub (256 MiB / disk)      | -    | 1.48 s (173 MB/s)  | 1.50 s (171 MB/s)           | **1.29 s (198 MB/s)** (+15%) |
| rebuild (256 MiB / disk)    | -    | 1.86 s (138 MB/s)  | 2.31 s (111 MB/s)           | **1.19 s (215 MB/s)** (+56%) |

**Highlights:**

- **kmec seq write +31% over raid6.**  The cross-stripe pipeline (each user
  bio sliced into per-stripe work items dispatched to system_unbound_wq
  independently, with per-stripe parallel child I/O within each) wins on
  large-block sequential writes — they're the workload that maps cleanly to
  the full-stripe write path.
- **kmec rand 4K write is slower than raid6** in absolute MB/s on this
  loop-on-a-single-virtio-file setup, but the structural cause (RAID4-
  style fixed parity placement) was fixed by adding parity rotation in
  a follow-on commit — see `notes/bench_analysis_2026-04-27.md`.  With
  rotation, per-disk util is now uniform at ~71-72 % across all six
  loops (vs the earlier 88 %/88 %/51 % split).  Throughput on this
  test setup is unchanged because all loops share one underlying
  virtio device — rotation moves the bottleneck off "fixed parity
  disks pinned at 88 %" but the system as a whole was already capped
  by the underlying virtio device's IOPS regardless.  On real
  independent disks the rotation should produce the expected ≈ 3×
  win; the test that would actually measure that needs k+m
  independent NVMe/SSD spindles.
- **kmec scrub +15 % and rebuild +56 % over raid6** in MB/s terms.
  Less state-machine transition overhead per stripe; sync_request
  goes straight from `fill_data_chunks` → encode → write.  Both still
  walk stripes single-threaded inside md_do_sync, so there's headroom
  if cross-stripe parallelism gets wired into sync_request too.
- **raid_isal.ko (GFNI override of raid6 syndrome) is ≈ stock raid6.**  On
  this single-virtio-loop setup, the GFNI math isn't the bottleneck — disk
  scheduling + memcpy are.  raid_isal's wins live elsewhere (real disks,
  high IOPS workloads, AVX-512 hosts).

## rhel10 VM (no-GFNI Zen 3, kernel 6.12.0-124.8.1.el10_1)

kmec needs GFNI and refuses to activate here — that gate itself is verified
in our test suite.  Instead we re-run stock raid6 vs our customised
raid456.ko (changes #1–#12, mainly stripe-cache + lock contention work) on
the same VM the README's existing benchmark numbers came from.

| Workload                    | bs   | A. stock raid6 | B. ours raid456 |
|-----------------------------|------|---------------:|----------------:|
| seq write (full-stripe)     | 256K | 2549 MB/s      | 2522 MB/s       |
| seq read                    | 256K | 6578 MB/s      | 6604 MB/s       |
| rand write (sub-stripe RMW) | 4K   | 215 MB/s       | 211 MB/s        |
| rand read                   | 4K   | 536 MB/s       | 546 MB/s        |
| scrub (252 MiB)             | -    | 0.42 s         | 0.42 s          |
| rebuild (256 MiB loop0)     | -    | 0.20 s         | 0.20 s          |

The two configs are identical within noise on this single-virtio-loop
setup.  Our raid456 changes target stripe-cache contention and rebuild
pipeline — improvements that show up under multi-disk + concurrent-IO
workloads (see the README's earlier 4-phase benchmark on a real virtio-disk
3-drive RAID-5).  On this 6-loop-on-one-file scenario the contention
patterns aren't exercised, so the deltas vanish into noise.

(Why are rhel10 numbers higher than rd07's?  The rhel10 raid6 array has
about 252 MiB of usable space due to the mdadm 1.2 superblock + bitmap
overhead on small loop devices, while rd07's superblock-less kmec gets
the full 1 GiB.  Smaller working set fits more comfortably in the page
cache layers below the loop device, so rhel10's "disk" reads are very
fast.)

## Caveat

These benchmarks measure CPU-bound EC work, lock overhead, and stripe
pipelining — **not** physical disk parallelism.  All six loops back to one
XFS file on a single virtio disk; per-stripe and cross-stripe parallel I/O
can't show their full benefit here.  On a real array of independent SSDs
or HDDs the disk-parallelism wins should multiply by k+m for the I/O-bound
workloads (writes, scrub, rebuild).  The numbers above show that the
correctness paths and CPU work are competitive with raid6 across a range
of shapes; the next benchmark to run is on real hardware.
